# ADX / ATR Regime Classification V2

Step 19 adds one canonical backend `RegimeState` produced by `AdxAtrRegimeClassifier`.

## Inputs

The classifier consumes the point-in-time feature snapshot generated by the shared feature engine:

- `spy1mAdx14`
- `spy1mAtr14`
- `spy1mRealizedVolatilityPercentile`
- `spy1mHigherHighHigherLow`
- `spy1mLowerHighLowerLow`
- completed `spy1mCandles` from `rawInputs`
- `economicEventState`

All measurements are derived from candles completed at or before the evaluation timestamp. If any required regime input is missing, malformed, or stale beyond the configured maximum feature age, the classifier returns `label=unknown`, `confidence=0`, `direction=0`, and `features.dataReady=false`.

## Outputs

The classifier returns `RegimeState` with these initial labels:

- `strong_trend`
- `weak_trend`
- `range`
- `low_volatility`
- `high_volatility`
- `event_shock`
- `unknown`

`RegimeState.direction` is a context-only directional bias inferred from current market structure. It must never substitute for a strategy `BUY` or `SELL` output.

The `features` payload stores separate per-family fit values:

- `trendFit`
- `breakoutFit`
- `reversalFit`
- `meanReversionFit`
- `gapSessionFit`

These fit values are normalized from `0.0` to `1.0` and are separate from `RegimeState.confidence`, which represents classifier certainty.

## Guardrails

- Mean-reversion fit is capped lower in `strong_trend` and `event_shock`.
- Trend and breakout fit are capped lower in low-directional `range` and `low_volatility` regimes.
- The classifier does not read session direction, event direction, or any proxy side field.
- `ADX Trend Strength Regime` and `ATR Volatility Regime` remain registered as regime modules only and cannot be directional voters.
